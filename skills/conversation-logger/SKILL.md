# Conversation Logger

## Purpose

Observe the current conversation and extract recurring patterns that indicate the user's working style, preferences, and common corrections. Write these patterns to a persistent log file for later analysis by the `skill-extractor` skill.

## What to Capture

Scan the conversation for these four pattern types:

| Pattern Type             | What to Look For                                                    | Example                                                         |
| ------------------------ | ------------------------------------------------------------------- | --------------------------------------------------------------- |
| **Repeated Instruction** | User restates something they likely said in previous conversations  | "Remember, always use English for commit messages"              |
| **Correction**           | User corrects AI behavior mid-conversation                         | "No, don't add Co-Author" / "I said discuss first, don't code" |
| **Preference Signal**    | User chooses option A over B, or expresses a preference             | Chose Hono over Express, prefers Cloudflare over AWS            |
| **Reusable Insight**     | A solution, workaround, or decision with cross-project value        | "AES-256-GCM with dual-key for zero-knowledge encryption"      |

## What NOT to Capture

- One-off factual Q&A with no pattern value
- Pure concept explanations the user asked about once
- Temporary debugging steps that won't recur
- Anything already recorded in AI Brain files

## Storage

### File Location

Path: see ai-brain Paths section. File: `$BRAIN_ROOT/_common/conversation-patterns.md`.

If the file does not exist, create it with this header:

```markdown
# Conversation Patterns Log

> Auto-generated by conversation-logger skill.
> Analyzed periodically by skill-extractor to identify Skill candidates.
```

### Entry Format

Append each entry at the end of the file:

```markdown
### YYYY-MM-DD | Topic: <brief topic description>
- **Repeated instruction:** <what the user restated>
- **Correction:** <what the user corrected and what the correct behavior is>
- **Preference:** <what the user chose or expressed preference for>
- **Insight:** <reusable technical knowledge>
```

Only include the pattern types that actually occurred. If a conversation only had a correction, only write the correction line. Do not pad with empty patterns.

## Execution Rules

### When to Execute

1. **End of conversation** — Before the final response, scan the full conversation and log any patterns found.
2. **On explicit trigger** — User says "log this", "記錄一下", "record this", or similar.
3. **On significant correction** — When the user corrects AI behavior, log the correction immediately (do not wait for conversation end).

### How to Execute

1. Scan the conversation history for the four pattern types.
2. Filter out anything that is not a genuine pattern (single occurrences with no reuse value).
3. Append to `conversation-patterns.md` using the entry format above.
4. Check if any reusable cross-project insight was learned → update the relevant `_common/` catalog file.
5. Notify the user briefly: "Logged [N] patterns to conversation-patterns.md" with a one-line summary of what was recorded.

### Self-Detection Logging

When `prompt-engineer` detects a potential violation mid-conversation but the user declines to act on it, `conversation-logger` should still record it as a pattern entry with a special tag:

```markdown
- **Self-detected:** [description of what AI noticed] (user declined to act)
```

This ensures the pattern is not lost and can be picked up by `skill-extractor` in the next analysis, even if the user chose not to address it immediately.

### Limits

- Maximum 5 pattern entries per conversation (keep signal-to-noise ratio high).
- If no patterns are detected, do not write anything — skip silently.
- Do not duplicate entries that are already in the log file. Check before appending.

## Integration with AI Brain

`conversation-patterns.md` is a Skill-owned file (see ai-brain Brain Structure). It does NOT need a `_catalog.json` entry — it is consumed by `skill-extractor`, not by the Brain loading procedure.
